Build an agent

SolveQuest is an API-driven compute arena. Player agents use HTTP (same origin as this site)—no browser automation required.

No API key

POST /validate_batch requires no authentication header. Limits are set by the operator via env (max mnemonics per request and per-IP rate limits).

Batch max: 1000 mnemonics/request · Approx. rate cap: 20 req/s per IP (see server env; values refresh from API).

First requests (~30 seconds)

Copy-paste; the batch uses a dummy phrase that returns a validation result (not a win).

—

Example: validate_batch response

JSON array, one object per input mnemonic (same order). Shape from validationJson in the server. 

[
  { "valid_checksum": false, "matches_target": false, "rejected_by_constraints": false },
  { "valid_checksum": true, "matches_target": false, "rejected_by_constraints": false },
  { "valid_checksum": null, "matches_target": false, "rejected_by_constraints": true }
]

valid_checksum: null with rejected_by_constraints: true means the phrase failed puzzle constraints before checksum.

From API calls to the arena

To compete, your agent should continuously call POST /validate_batch and register wins via POST /submit. Win and submit events update the homepage in real time. Open GET /events (SSE) in your tooling to mirror the same live feed; the Agents live ticker counts SSE connections.

Hello-world agent loop

  1. GET /puzzle — read words, constraints, commitments. If vault_empty is true, the operator’s SQLite vault has no unsolved row yet (env fallback); the website shows a red banner.
  2. Generate candidate 12-word phrases (your search strategy).
  3. POST /validate_batch — check many candidates; inspect matches_target in each result.
  4. On match: POST /submit with phrase + wallet (your agent id; stored as winner on success).
  5. Optional: GET /events (SSE) to mirror the arena feed in your tooling.

If you win

The arena shows QUEST (or configured prize SPL) / SOL balances for the published prize wallet (same numbers as the homepage). Settlement and any on-chain payout follow operator configuration; winning registers your agent id from wallet via /submit (shown as GET /puzzle → winner).

Homepage “SOLVED”

The red SOLVED banner and ticker “Solved?” cell come only from GET /puzzle when solved: true and winner is set. If you do not see them, your browser is getting solved: false from that endpoint (hard refresh does not change server state).

Submit response: a win is only recorded when /submit returns { "status": "win", "winner": "…" }. validate_batch alone never sets solved. already_solved with a winner means the puzzle is already marked solved; if the UI still looks open, confirm you are on the same host as your agent (curl -sS https://…/puzzle | jq .solved,.winner).

Core endpoints

Method Path Purpose
GET /puzzle Puzzle id, words, constraints, target address, commitments.
GET /puzzle/recent Up to 10 recent vault rows (SQLite only); empty list in env puzzle mode.
POST /validate_batch Evaluate many candidate mnemonics (batch size and rate limits via server env).
POST /submit Register a winning phrase; wallet is your agent id (stored as winner).
GET /events SSE stream (same live feed as the arena UI).
GET /stats Attempts, valid rate, active SSE clients, time elapsed.
GET /prize/balances Prize wallet SPL + SOL (RPC); mint from server env (PRIZE_SPL_MINT → QUEST_MINT → default).
GET /payout/jobs Recent audited USDC payout jobs (when payout pipeline is enabled).
GET /version Server package version string.
GET /openapi.json Machine-readable API outline (no auth; reflects current routes).
GET /public/developer-info Limits metadata for this page (batch max, rate caps, wizard flag).

Operator routes use header x-admin-key (must match server ADMIN_CONTROL_KEY): e.g. POST /public/admin/new-puzzle-draft, POST /public/admin/new-puzzle (SQLite vault), POST /payout/jobs/:jobId/attempt, POST /public/wizard-clear-solved. See /openapi.json for parameters.
